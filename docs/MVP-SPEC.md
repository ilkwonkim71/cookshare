# CookShare MVP 기능 명세서

레시피 공유 서비스의 MVP(최소 기능 제품) 명세입니다. 사용자 스토리 형식으로 기술하며,
우선순위(P0/P1/P2)와 예상 구현 시간을 포함합니다.

- 작성일: 2026-06-17
- 대상 스택: Next.js 14 / Express + TS / SQLite / JWT / 로컬 이미지 저장

## 우선순위 정의

| 우선순위 | 의미                  | 설명                                                      |
| -------- | --------------------- | --------------------------------------------------------- |
| **P0**   | Must-have (출시 필수) | 이게 없으면 서비스가 성립하지 않음. MVP 1차 릴리스 범위.  |
| **P1**   | Should-have (중요)    | 사용성/완성도를 크게 높임. 1차 직후 빠르게 따라붙는 범위. |
| **P2**   | Nice-to-have (선택)   | 있으면 좋지만 없어도 핵심 흐름에 지장 없음. 여유 시 진행. |

> 예상 시간은 **1인 풀스택 개발자 기준 순수 구현 시간**(설계+코딩+기본 테스트)이며,
> FE/BE를 합산한 값입니다. 리뷰/QA/배포 시간은 제외합니다.

## 1. 인증 & 계정 (Auth)

### US-1.1 회원가입 — **P0** · 4h

> **As a** 방문자
> **I want** 이메일·비밀번호·이름으로 가입하고 싶다
> **So that** 내 레시피를 등록하고 관리할 수 있다.

- **인수 조건**
  - 이메일/비밀번호/이름 입력 폼 제공
  - 이메일 형식 검증, 중복 이메일 시 명확한 에러
  - 비밀번호는 bcrypt로 해시되어 저장 (평문 저장 금지)
  - 성공 시 JWT 발급 + 자동 로그인 상태로 홈 이동
- **상태**: 백엔드 `POST /auth/register` 구현됨, 프론트 `/register` 폼 존재

### US-1.2 로그인 / 로그아웃 — **P0** · 3h

> **As a** 가입한 사용자
> **I want** 로그인/로그아웃 하고 싶다
> **So that** 내 계정으로 작업하고 안전하게 종료할 수 있다.

- **인수 조건**
  - 잘못된 자격 증명 시 일반화된 에러 메시지("이메일 또는 비밀번호가 올바르지 않습니다")
  - 로그인 성공 시 토큰을 클라이언트에 저장, 헤더 UI가 로그인 상태로 전환
  - 로그아웃 시 토큰 폐기 및 비로그인 UI 복귀
- **상태**: `POST /auth/login`, `GET /auth/me`, 프론트 AuthContext 구현됨

### US-1.3 세션 유지 & 보호 라우트 — **P1** · 3h

> **As a** 로그인한 사용자
> **I want** 새로고침해도 로그인이 유지되길 원한다
> **So that** 매번 다시 로그인하지 않아도 된다.

- **인수 조건**
  - 새로고침 후에도 토큰으로 세션 복원 (`/auth/me`)
  - 비로그인 상태에서 작성 페이지 접근 시 로그인으로 리다이렉트
  - 토큰 만료/무효 시 자동 로그아웃 처리

### US-1.4 비밀번호 재설정 — **P2** · 8h

> **As a** 비밀번호를 잊은 사용자
> **I want** 이메일로 비밀번호를 재설정하고 싶다
> **So that** 계정 접근을 복구할 수 있다.

- **인수 조건**: 재설정 토큰 발급, 메일 발송(개발 환경은 콘솔/스텁), 새 비밀번호 설정
- **비고**: 메일 인프라 필요 → MVP 범위 밖, 추후

## 2. 레시피 작성 & 관리 (Authoring)

### US-2.1 레시피 등록 — **P0** · 6h

> **As a** 로그인한 사용자
> **I want** 제목·설명·재료·조리 단계·조리시간·인분으로 레시피를 등록하고 싶다
> **So that** 내 요리법을 공유할 수 있다.

- **인수 조건**
  - 재료/조리 단계는 여러 항목을 동적으로 추가·삭제 가능
  - 필수값(제목, 재료 1개 이상, 단계 1개 이상) 검증
  - 등록 성공 시 상세 페이지로 이동
- **상태**: `POST /recipes`, 프론트 `/recipes/new` 동적 폼 구현됨

### US-2.2 이미지 업로드 — **P0** · 5h

> **As a** 레시피 작성자
> **I want** 대표 이미지를 첨부하고 싶다
> **So that** 레시피를 시각적으로 매력 있게 보여줄 수 있다.

- **인수 조건**
  - 이미지 파일만 허용, 5MB 용량 제한, 초과/비이미지 시 에러
  - 업로드 → URL 수신 → 레시피의 `imageUrl`로 연결
  - 로컬 디스크 저장, `/uploads/*`로 정적 제공 (S3 전환 추상화 유지)
- **상태**: `POST /uploads`, Storage 추상화, 프론트 업로드 플로우 구현됨

### US-2.3 레시피 수정 — **P1** · 4h

> **As a** 레시피 작성자
> **I want** 내가 올린 레시피를 수정하고 싶다
> **So that** 오타나 내용을 고칠 수 있다.

- **인수 조건**
  - 작성자 본인만 수정 가능 (타인 403)
  - 기존 값이 채워진 수정 폼, 이미지 교체 가능
  - `PUT /recipes/:id`
- **상태**: 백엔드 구현됨, **프론트 수정 페이지 미구현** (`/recipes/[id]/edit`)

### US-2.4 레시피 삭제 — **P1** · 2h

> **As a** 레시피 작성자
> **I want** 내 레시피를 삭제하고 싶다
> **So that** 더 이상 공유하고 싶지 않은 글을 내릴 수 있다.

- **인수 조건**
  - 작성자 본인만 삭제 가능 (타인 403)
  - 삭제 전 확인 다이얼로그
  - 삭제 후 목록으로 이동
- **상태**: `DELETE /recipes/:id`, 상세 페이지 삭제 버튼 구현됨 (확인 다이얼로그는 보강 필요)

### US-2.5 임시 저장(Draft) — **P2** · 6h

> **As a** 작성자
> **I want** 작성 중인 레시피를 임시 저장하고 싶다
> **So that** 나중에 이어서 완성할 수 있다.

- **인수 조건**: 공개/비공개(draft) 상태, 본인만 draft 조회
- **비고**: `status` 컬럼 추가 필요, MVP 이후

## 3. 레시피 탐색 & 조회 (Discovery)

### US-3.1 레시피 목록 — **P0** · 4h

> **As a** 방문자
> **I want** 등록된 레시피를 카드 그리드로 둘러보고 싶다
> **So that** 관심 있는 요리를 찾을 수 있다.

- **인수 조건**
  - 대표 이미지·제목·작성자·조리시간을 카드로 표시
  - 페이지네이션(페이지당 20개)
  - 비로그인도 열람 가능
- **상태**: `GET /recipes?page=`, 홈 그리드 구현됨

### US-3.2 레시피 상세 — **P0** · 4h

> **As a** 방문자
> **I want** 레시피 상세(재료·단계·정보)를 보고 싶다
> **So that** 실제로 요리를 따라 할 수 있다.

- **인수 조건**
  - 이미지, 재료 목록, 번호 매겨진 조리 단계, 작성자/조리시간/인분 표시
  - 존재하지 않는 레시피는 404
- **상태**: `GET /recipes/:id`, 상세 페이지 구현됨

### US-3.3 검색 — **P1** · 3h

> **As a** 방문자
> **I want** 키워드로 레시피를 검색하고 싶다
> **So that** 원하는 요리를 빠르게 찾을 수 있다.

- **인수 조건**
  - 제목/설명 기준 부분 일치 검색
  - 검색어 유지 + 결과 없을 때 빈 상태 UI
  - `GET /recipes?q=`
- **상태**: 백엔드 검색 지원, 프론트 검색창 존재 (결과 없음 UI 보강 권장)

### US-3.4 카테고리/태그 필터 — **P2** · 6h

> **As a** 방문자
> **I want** 카테고리(한식/양식/디저트 등)나 태그로 필터링하고 싶다
> **So that** 취향에 맞는 레시피만 볼 수 있다.

- **비고**: `categories`/`tags` 스키마 추가 필요, MVP 이후

## 4. 사용자 프로필 & 상호작용 (Social)

### US-4.1 작성자별 레시피 보기 — **P1** · 3h

> **As a** 방문자
> **I want** 특정 작성자의 레시피 목록을 보고 싶다
> **So that** 마음에 든 작성자의 다른 요리도 볼 수 있다.

- **인수 조건**: `/users/:id` 또는 `?authorId=` 필터, 프로필 헤더(이름/레시피 수)

### US-4.2 내 레시피 관리 (마이페이지) — **P1** · 4h

> **As a** 로그인한 사용자
> **I want** 내가 올린 레시피를 한곳에서 보고 수정/삭제하고 싶다
> **So that** 내 콘텐츠를 효율적으로 관리할 수 있다.

- **인수 조건**: `/me` 페이지, 내 레시피 목록 + 수정/삭제 진입점

### US-4.3 좋아요(찜) — **P2** · 5h

> **As a** 로그인한 사용자
> **I want** 마음에 드는 레시피에 좋아요를 누르고 싶다
> **So that** 나중에 다시 찾아보거나 인기 레시피를 알 수 있다.

- **비고**: `likes` 테이블 추가, MVP 이후

### US-4.4 댓글 — **P2** · 6h

> **As a** 로그인한 사용자
> **I want** 레시피에 댓글을 달고 싶다
> **So that** 후기나 질문을 남길 수 있다.

- **비고**: `comments` 테이블 추가, MVP 이후

## 5. 공통 / 비기능 (Cross-cutting)

### US-5.1 반응형 레이아웃 — **P0** · 3h

> **As a** 모바일/데스크톱 사용자
> **I want** 어느 기기에서나 잘 보이길 원한다
> **So that** 주방에서 휴대폰으로도 레시피를 볼 수 있다.

- **상태**: shadcn/Tailwind 기반 반응형, 기본 충족 (QA 필요)

### US-5.2 입력 검증 & 에러 처리 — **P0** · 4h

> **As a** 사용자
> **I want** 잘못된 입력에 명확한 안내를 받고 싶다
> **So that** 무엇을 고쳐야 할지 알 수 있다.

- **인수 조건**: FE/BE 양쪽 검증(zod), 일관된 에러 응답 형식, 토스트/인라인 메시지
- **상태**: 백엔드 zod + 에러 핸들러, 프론트 토스트 구현됨

### US-5.3 로딩 & 빈 상태 — **P1** · 3h

> **As a** 사용자
> **I want** 로딩 중/데이터 없음을 분명히 보고 싶다
> **So that** 멈춘 건지 비어있는 건지 헷갈리지 않는다.

- **인수 조건**: 스켈레톤/스피너, 빈 목록·빈 검색 결과 안내 UI

### US-5.4 기본 보안 하드닝 — **P1** · 4h

> **As a** 서비스 운영자
> **I want** 기본적인 보안 조치를 적용하고 싶다
> **So that** 흔한 공격으로부터 사용자를 보호할 수 있다.

- **인수 조건**: helmet, rate limiting(인증/업로드), CORS 화이트리스트, JWT 만료, 업로드 MIME/용량 재검증

### US-5.5 배포 준비 — **P2** · 6h

> **As a** 운영자
> **I want** 스테이징/프로덕션에 배포하고 싶다
> **So that** 실제 사용자에게 서비스를 제공할 수 있다.

- **비고**: 환경별 설정, SQLite→Postgres 검토, 로컬→S3 전환, CI 파이프라인

## 현재 진척도 메모

스캐폴딩 단계에서 다음 P0/P1 항목의 **기반 구현이 이미 존재**합니다(QA·보강 필요):

- ✅ 회원가입/로그인, JWT, AuthContext (US-1.1, 1.2)
- ✅ 레시피 등록/목록/상세, 이미지 업로드, Storage 추상화 (US-2.1, 2.2, 3.1, 3.2)
- ✅ 삭제 API + 버튼, 검색 API + 검색창 (US-2.4, 3.3)
- ⛏️ 미구현/보강 대상: 레시피 **수정 페이지(FE)**(US-2.3), 보호 라우트 리다이렉트(US-1.3),
  마이페이지(US-4.2), 삭제 확인 다이얼로그, 빈 상태 UI, 보안 하드닝(US-5.4)

따라서 P0를 "완성도 있는 1차 릴리스" 수준으로 마감하는 데 실제 잔여 작업은
명세 추정치(33h)보다 적습니다(대략 10~15h: 검증/QA + 수정 페이지 + 보호 라우트 + 에러/빈 상태 다듬기).
